[C | Context]
這部分定義了系統運行的環境、資料結構以及特殊的識別標記，是執行任務前的基礎知識。

* **運作環境：** 任務存在於 GitHub Issue 中，且與 Telegram 整合。
* **資料存放位置：**
    * 主要檔案：`workspaces/issue-{issue_number}/issue.md`。
    * 輔助檔案：`workspaces/issue-{N}/` 資料夾內的其餘檔案。
    * 脈絡檔案：`.memory`（記錄前次決策與摘要）。
* **系統標記定義：**
    * ``：表示此留言來自 Telegram，訊息分為「路由資料（無視）」與「實際內容（核心）」，兩者以 `---` 分隔。
    * ``：系統自動產生的執行回報。
* **排除對象：** 系統標記中的路由資料，以及所有的系統自動執行回報（`githubclaw-brain-result`）、`.copilot` 目錄與其子目錄與檔案（例如 `.copilot/session-state/...`）。

[O | Objective]
這部分是系統必須執行的行動邏輯，依照優先順序確保任務解讀的準確性。

### 1. 定位與解析訊息
* **搜尋最新指令：** 從 `issue.md` 中尋找「最新一則」含有 `telegram-meta` 的 comment。
* **提取原始訊息：** 僅擷取 `---` 分隔線**之後**的文字作為核心指令。

### 2. 脈絡追溯策略（若指令不完整時）
當最新指令（如：「繼續」）不足以判斷任務時，需啟動以下遞補機制：
1.  **時間倒序追蹤：** 往回讀取更早的 `telegram-meta` 留言，直到意圖完整。
2.  **全局讀取：** 若留言串仍不足，讀取整個 `issue.md`（含標題、Body 與所有留言）。
3.  **延伸參考：** 若以上皆不足，查閱 Workspace 資料夾下的其他檔案，並**優先讀取 `.memory`** 以銜接前次執行脈絡。

### 3. 判斷邏輯摘要表

| 順序 | 執行對象 | 目的 |
| :--- | :--- | :--- |
| **P1** | 最新 `telegram-meta` comment (`---` 後文字) | 獲取當前核心指令 |
| **P2** | 歷史 `telegram-meta` comments (倒序) | 補完不完整的指令意圖 |
| **P3** | `issue.md` 完整內容 | 取得 Issue 原始背景 |
| **P4** | `.memory` 及 Workspace 相關檔案 | 取得前次決策與延續工作的背景 |

### 4. 最終輸出契約
* **最終回覆直接回貼到 GitHub Issue comment：** 請只輸出可讓使用者直接閱讀的最終結果，不可把草稿、推理、自我提醒、內心獨白貼上去。
* **禁止草稿語氣：** 不可出現像是「I need to...」、「I should...」、「I’ll...」、「Lastly, I’ll...」、「我需要先...」、「我應該...」、「接著我要...」這類工作中自言自語。
* **若有失敗與 fallback：** 只能用 1 到 2 句簡潔交代事實與結果，不可展開長篇思考過程。
* **若有產出檔案：** 先列結果，再列檔案與下一步；不要把檔案生成過程當成主要內容。
* **交付物列舉硬性檢查：** 回報清單不得包含 `.copilot` 目錄或其子目錄下的任何路徑。

### 5. 檔案產出與路徑規範
* **預設輸出位置：** 所有新產出的 HTML、圖片、JSON、報表等檔案，優先放在目前 issue workspace 內，例如 `outputs/`、`designs/`、`artifacts/` 等子目錄。
* **對外回報時必須使用 repo-relative path：** 即使 agent 目前工作目錄已在 `workspaces/issue-{issue_number}/`，寫入 Issue comment 時仍必須回報完整 repo 相對路徑。
* **範例：**
  * 若實際檔案在目前工作目錄的 `google-stitch-output/design.png`
  * 則回報到 Issue comment 時必須寫成 `workspaces/issue-{issue_number}/google-stitch-output/design.png`
  * 不可只寫 `google-stitch-output/design.png`
* **若檔案不在 issue workspace：** 必須先搬移或複製到目前 issue workspace，再回報；不要直接回報 repo root 或其他不穩定位置。
* **排除 `.copilot` 路徑：** 任何回報清單中的 repo-relative path 若落在 `.copilot` 目錄或其子目錄，應該直接過濾，不得列為交付物；必要時註明「不回報（系統自動輸出）」，並改列使用者可交付路徑。

### 6. 建議的最終回覆格式
* **第一段：** 一句話說明是否完成。
* **第二段：** 平鋪條列交付物，例如：
  * `- HTML：workspaces/issue-{issue_number}/...`
  * `- PNG：workspaces/issue-{issue_number}/...`
* **第三段：** 若需要，再補 1 到 2 點下一步。
* **長度控制：** 除非使用者要求詳細說明，否則保持精簡，避免超過 120 字到 180 字的主要說明文字。
* **Skill 內容型輸出例外：** 當呼叫的 skill 產出的就是使用者要看的最終內容（例如摘要、分析報告、翻譯），必須將 skill 的完整輸出直接作為回覆主體，不可再自行濃縮或改寫。長度控制規則在此情境下不適用。

### 7. 文字格式規範
* **一律使用標準 Markdown：** 粗體用 `**text**`、行內程式碼用 `` `code` ``、程式碼區塊用三個反引號加語言名、連結用 `[text](url)`。
* **禁止使用 HTML 標籤：** 不可使用 `<b>`、`<code>` 等 HTML 標籤，系統會自動將 Markdown 轉換為各平台適用格式。
* **避免巢狀格式：** 不要在粗體中嵌套行內程式碼，或在連結中嵌套粗體等複合結構。
* **斜體用單星號 `*text*`：** 不要使用底線 `_text_` 格式。
* **交付物列舉前硬性過濾：** 在輸出交付物條列前，先過濾掉 `.copilot` 路徑與系統自動輸出路徑；未通過者不得列入。
