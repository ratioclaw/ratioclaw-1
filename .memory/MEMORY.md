# Repository Memory

## Stable Context
- **技術決策須保留工程師的專業判斷**  
  在產品會議出現突發「緊急需求」時，資深工程師會立即指出實際所需的時間與測試資源。此類情況必須以工程師的估算為基礎，避免因時間壓力而妥協設計品質。

- **知識文件化是團隊風險管理的核心**  
  - 已開始編寫 **ADR（Architecture Decision Record）**、**Runbook**、以及 **系統圖**，目前已完成約一年份的內容。  
  - 團隊正持續撰寫「不可或缺的人是團隊風險」的案例故事，以提醒成員避免知識孤島。  
  - Legacy 系統中僅有 **柏宇** 能解讀的「魔法 SQL」被視為高風險點，必須納入文件化流程。

- **四項持續推進的 Active Tasks**（自 2026‑04‑10 起持續）  
  1. 完成上述文件化工作（ADR、Runbook、系統圖）。  
  2. 撰寫「不可或缺的人是團隊風險」的敘事稿。  
  3. 追蹤與解決產品會議中突發的緊急需求。  
  4. 監控測試資源與時間分配，確保不因緊急需求而犧牲測試完整性。

- **團隊協作慣例**  
  - 所有 Issue 必須在 **30 天內** 重新評估其狀態。  
  - 標籤 `chat-session` 用於記錄即時討論與決策過程。  
  - 每日快照由多個 **issue agents** 產出，若無可用 Issue，則保留既有記憶不變。
