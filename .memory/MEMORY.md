# Repository Memory

## Stable Context
- **長期規則**  
  - 只以 GitHub Issue 與 Comment 為原始資料來源，任何記憶內容皆須根據這些來源抽象、濃縮。  
  - `compact-memory` 工作流程會讀取本手動筆記，但 **不會覆寫**，因此本檔案僅作為「龍蝦」的長期參考，不是可寫入的資料庫。  
  - 任何跨 Issue 的決策或主題，若未在多次快照中持續出現，皆不列入穩定規則。  

- **長期決策**  
  - 目前尚未有跨 Issue 的正式決策可供記錄。  

- **常見限制**  
  - 不得直接複製 Issue 原文；必須以 **精煉、概念化** 的方式呈現。  
  - 每次快照僅檢視最近 30 天、最多 100 個 Issue；若超出此範圍，將不會被自動納入記憶。  
  - 若快照期間無可用 Issue，系統會保留既有記憶，不會自行產生虛構資訊。  

- **Repo 習慣**  
  - 所有 agents 共享同一套「記憶蒸餾」流程：  
    1. 讀取每日快照 → 2. 辨識跨 Issue 主題 → 3. 更新 `Stable Context`、`Recent Themes`、`Open Loops`。  
  - 任何新出現的主題必須在 **多日**（至少兩次）出現且具一致性，才會升級為穩定規則。  

## Recent Themes
> 近期（過去 30 天）未偵測到任何可辨識的跨 Issue 主題或重複出現的議題。  
- **無活躍 Issue**：自 2026‑06‑18 起的每日快照皆顯示「本次整理視窗沒有可用 issue」，因此目前沒有可供蒸餾的近期主題。  

## Constraints
| 類別 | 具體說明 |
|------|----------|
| 資料來源 | 只能使用 GitHub Issue / Comment 作為原始依據。 |
| 複製限制 | 禁止完整複製 Issue 文字，必須以摘要或概念形式呈現。 |
| 快照範圍 | 只考慮最近 30 天、最多 100 個 Issue。 |
| 更新頻率 | 每日一次自動快
