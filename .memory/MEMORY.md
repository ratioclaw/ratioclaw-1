# Repository Memory

## Stable Context
- **資料來源**：長期記憶的原始依據必須是 GitHub Issue 或 Comment，手動維護的 `shared/manual.md` 只作為輔助說明，**不會**被自動覆寫。  
- **記憶更新原則**：每日快照只在有可用 Issue 時才會產生新資訊；若當天無 Issue，則保留既有記憶，不新增或刪除內容。  
- **共通慣例**：所有 agents 必須遵守 repo 的共通規範，避免直接複製 Issue 原文，僅抽取可重用的概念或決策。  

> **不確定性**：目前尚未觀測到任何持續出現的規則、決策或限制，以上為唯一可確認的長期穩定資訊。

## Recent Themes
- **缺乏活躍 Issue**：連續 7 天（2026‑07‑08 至 2026‑07‑14）皆未偵測到可用 Issue，導致每日快照皆顯示「保留既有記憶」且無跨 Issue 主題。  
- **等待更新**：每日快照的「Open Loops」皆指向「等待下一輪 issue 更新後再整理」，顯示目前的工作焦點在於監控 Issue 流入，而非產出新決策。  

> **不確定性**：若未來出現新 Issue，可能會形成新的主題或決策，需在下一次快照中重新評估。

## Constraints
1. **來源限制**：只能從 GitHub Issue / Comment 抽取資訊，手動筆記僅供參考。  
2. **內容限制**：不得完整複製 Issue 原文，必須以精煉、可重用的形式呈現。  
3. **更新頻率**：每日快照只在「自上次快照以來 30 天內」且有 Issue 時才會產生實質內容。  
4. **手動筆記保護**：`shared/manual.md` 由人類維護，系統不會自動覆寫或刪除其中的條目。  

## Open Loops
- **Issue 待入**：目前沒有任何待處理的 Issue，所有 agents 正在等待新 Issue 的產生。  
- **跨 Issue 主題探索**：缺乏可辨識的跨 Issue 主題，未來若出現多個相關 Issue，需建立相應的主題彙整。  
- **決策記錄**：尚未有跨 Issue 的決策產出，需在有新 Issue 時開始追蹤與記錄。
