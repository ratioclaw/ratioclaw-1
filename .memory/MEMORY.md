# Repository Memory

## Stable Context
- **資料來源**：長期記憶的原始依據必須是 GitHub Issue 或 Comment，手動維護的 `shared/manual.md` 只作為補充說明，**不會被自動覆寫**。  
- **記憶更新機制**：每日快照會從最近 30 天內的活躍 Issue 中抽取資訊，若無可用 Issue，則保留既有記憶不變。  
- **Repo 習慣**：所有 agents 必須遵守「compact‑memory workflow」：  
  1. 只讀取 `shared/manual.md` 中的手動筆記。  
  2. 不直接複製 Issue 原文，而是蒸餾成可重用的上下文。  
  3. 若發現跨 Issue 的共通主題或決策，立即寫入本節的 **Stable Context**。  

> **目前觀測**：過去一週（2026‑07‑07 至 2026‑07‑13）內，倉庫中沒有任何可用的 Issue，故上述規則仍是唯一可確認的穩定事實。

## Recent Themes
- **無可辨識主題**：每日快照皆回報「目前沒有可辨識的跨 issue 主題」。  
- **暫無新決策**：同樣回報「目前沒有新的跨 issue 決策」。  

> **說明**：因缺乏 Issue，近期未產生任何可歸納的主題或趨勢。

## Constraints
1. **資訊完整性**：若 Issue 資料缺失或不完整，agents 必須保留先前的記憶，避免產生空洞或錯誤的上下文。  
2. **手動筆記限制**：`shared/manual.md` 只能包含 **穩定規則、長期決策、常見限制、repo 習慣**，不應直接貼上 Issue 內容。  
3. **更新頻率**：每日快照僅檢視最近 30 天內的 Issue，超出此範圍的資訊不會自動納入本記憶檔。  

> **不確定性**：目前尚未觀測到任何具體限制或衝突情形，若未來出現相互矛盾的規則，需在此節標註「不確定」並說明衝突點。
