# Repository Memory

## Stable Context
- **資料來源原則**  
  - 所有長期記憶皆應以 GitHub Issue / Comment 為唯一原始資料來源。  
  - `shared/manual.md` 只作為人類手動維護的補充說明，**不會**被自動覆寫。  
- **Compact‑Memory 工作流程**  
  - 每日快照會從最近 30 天內的 Issue 中抽取資訊，蒸餾成「跨 Issue 主題」與「決策」等結構。  
  - 若快照期間無可用 Issue，系統會保留既有記憶並標示「等待下一輪 Issue 更新」。  
- **Repo 習慣**  
  - 以 **deterministic** 方式產生摘要，確保同一天的快照在相同輸入下產出相同結果。  
  - 每日快照上限 100 個 Issue，涵蓋所有狀態（open / closed），以避免遺漏長期未關閉的討論。  

> **不確定性**：目前缺乏任何 Issue 內容，無法抽取具體的穩定規則、決策或跨 Issue 主題。上述「資料來源原則」與「工作流程」為唯一可確認的長期規則。

## Constraints
1. **避免原文複製**：任何記錄皆須經過蒸餾、重組，不能直接貼上 Issue 標題或內容。  
2. **資訊完整性**：若資訊不足或相互矛盾，必須在此節明確標示「不確定」或「缺乏資料」。  
3. **工程可讀性**：使用 Markdown 結構、清晰的子標題與條列，方便自動化工具與人類閱讀。  
4. **更新頻率**：每日快照若持續無 Issue，需在「Open Loops」中提醒等待新 Issue 的產生。
