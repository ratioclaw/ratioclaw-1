# Repository Memory

## Stable Context
- **記憶管理原則**  
  - 只有 **GitHub Issue / Comment** 被視為原始資料來源，其他摘要或快照僅作為二次加工的參考。  
  - `shared/manual.md` 為 **手動維護的長期記憶**，永遠不會被自動覆寫，且不應直接複製完整 Issue 內容。  
  - `compact‑memory workflow` 會讀取此手動筆記以取得穩定規則與限制，然後再把每日快照的精華蒸餾進 `MEMORY.md`。  

- **Repo 內部慣例**  
  - 每日快照只會檢視最近 **30 天**、最多 **100 個** Issue。  
  - 若當日無可用 Issue，快照會保留既有記憶，不會產生新條目。  
  - 跨 Issue 的主題、決策與開放環節（Open Loops）皆需在有實際 Issue 時才會被記錄。  

- **長期決策與規則**（截至目前可確認）  
  - **不在長期記憶中直接保存 Issue 原文**，只保留概念與結論。  
  - **所有穩定規則皆寫於 `shared/manual.md`**，此檔案是唯一的「官方」長期記憶來源。  

> **不確定性**：目前未觀測到任何具體的業務流程、技術決策或團隊慣例超出上述手動筆記的描述，故無法補充其他穩定事實。

## Recent Themes
- **缺乏活躍 Issue**：從 2026‑06‑25 至 2026‑07‑01 的每日快照皆顯示「本次整理視窗沒有可用 issue」，說明近期（過去一週）專案內部沒有新 Issue 被建立或更新。  
- **無跨 Issue 主題**：每日報告均未偵測到可辨識的跨 Issue 主題或決策。  

> **觀察**：若未來持續缺少 Issue，可能代表開發節奏放緩、需求已穩定，或是 Issue 管理流程暫時停擺。此情況應在下一輪快照出現新 Issue 時重新評估。
