# Repository Memory

## Stable Context
- **記憶管理原則**  
  - 只從 GitHub Issue / Comment 抽取原始資訊，手動筆記 (`shared/manual.md`) 用來保存 **穩定規則、長期決策、常見限制**，且 **不會被自動流程覆寫**。  
  - `compact‑memory` 工作流會讀取此手動筆記作為全域上下文，決不直接複製完整 Issue 內容。  
- **Repo 習慣**  
  - 所有 agents 必須遵守「先保留既有記憶」的策略：若當前整理窗口找不到可用 Issue，則不新增或刪除任何條目。  
  - 每日快照 (`daily/*.json`) 只在有可用 Issue 時才會產生跨 Issue 主題、決策與待辦。  
- **限制**  
  - 只能在過去 **30 天** 內的 Issue 中搜尋；超出範圍的資訊不會自動納入快照。  
  - 每次快照最多處理 **100 個 Issue**，超過上限的會被截斷。  
- **決策流程**  
  - 跨 Issue 的決策必須在快照的 **Decisions** 區塊明確列出，否則視為「無新決策」。  

## Recent Themes
- **缺乏可用 Issue**  
  - 2026‑07‑15 至 2026‑07‑21 的每日快照皆顯示「本次整理視窗沒有可用 issue」，代表最近一週內 repository 沒有活躍的 Issue 需要被整理。  
- **待更新的記憶**  
  - 每日快照的 **Open Loops** 均寫為「等待下一輪 issue 更新後再整理」，顯示目前的主要待辦是 **等待新 Issue 的產生**。  

> **註**：因為連續七天皆無跨 Issue 主題或決策，無法從日誌中萃取出其他重複出現的主題。

## Constraints
1. **資訊來源限制**  
   - 只能使用 GitHub Issue / Comment 作為原始資料；手動筆記僅作為輔助說明。  
2. **內容抽取原則**  
   - 不得直接複製 Issue 原文，必須以 **摘要/蒸餾** 形式呈現。
