# Repository Memory

## Stable Context
- 目前的倉庫在過去 30 天內未產生任何可供分析的 Issue，故無法從實際工作流中萃取出穩定的規則或長期決策。  
- 依照 **Shared Memory Manual Notes**，以下原則被視為長期穩定的 repo 習慣：
  - 只以 GitHub Issue / Comment 作為原始資料來源，其他形式的筆記僅作為輔助說明。  
  - `shared/manual.md` 內的手動筆記 **不會** 被自動流程覆寫，僅供 agents 參考。  
  - agents 在整理記憶時 **不得** 完整複製 Issue 原文，需以精煉、抽象的方式呈現。  

> **不確定性**：除上述手動筆記外，缺乏實際 Issue 內容，使得目前無法確認其他潛在的穩定規則（如分支命名慣例、測試流程等）。

## Recent Themes
- 最近 30 天的每日快照皆顯示「本次整理視窗沒有可用 issue」，因此沒有可辨識的跨 Issue 主題或重複出現的討論焦點。  
- 由於缺乏新議題，近期也未產生新的決策或主題趨勢。

## Constraints
1. **資料來源限制**  
   - 只能從 GitHub Issue 與其 Comment 中抽取資訊。  
   - 任何非 Issue 的資訊（如聊天記錄、外部文件）不應直接納入自動記憶整理。  

2. **手動筆記保護**  
   - `shared/manual.md` 為手動維護的長期記憶，系統不會自動修改或覆寫此檔。  

3. **資訊抽象原則**  
   - agents 必須避免逐段複製原始 Issue 文字，應將資訊蒸餾為可重用的上下文描述。  

4. **更新頻率**  
   - 只在有新 Issue 產生或現有 Issue 有變動時才觸發記憶更新。  

## Open Loops
- **等待 Issue 更新**：目前所有每日快照皆顯示「等待下一輪 issue 更新後再整理」，因此以下項目仍未解決或持續開放：
  - 是否有即將產生的 Issue 會帶來新的跨 Issue 主題或決策。  
  - 任何潛在的長期規則或限制，需待新 Issue 出現後再行
