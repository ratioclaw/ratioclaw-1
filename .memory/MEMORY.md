# Repository Memory

## Stable Context
- **資料來源原則**：所有可追溯的事實皆必須來自 GitHub Issue 或 Comment，手動筆記僅作為彙整與提醒之用，且不會被自動流程覆寫。  
- **記憶工作流程**：`compact‑memory` 會讀取 `shared/manual.md` 作為長期記憶的基礎，並以每日快照 (`daily/*.json`) 逐步蒸餾新資訊。  
- **快照設定**：每日快照會檢索最近 30 天內、最多 100 個 Issue，使用 deterministic 演算法產生摘要。  
- **跨日一致性**：若同一天內無可用 Issue，系統會保留既有記憶，不會自行產生新條目。  

> **不確定性**：目前尚未在 Issue 中觀測到任何具體規則、決策或持續性主題，故上述為唯一可確認的長期規則。

## Recent Themes
- **無可辨識主題**：2026‑06‑01 至 2026‑06‑07 的所有快照皆顯示「本次整理視窗沒有可用 issue」，因此近期未出現重複或新興的議題。  

> 若未來出現持續出現的關鍵字或標籤，將在此節列出。

## Constraints
- **Issue 取樣上限**：每次快照最多處理 100 個 Issue。  
- **時間範圍**：僅考慮最近 30 天內的 Issue。  
- **摘要來源**：使用 deterministic 方法產生摘要，避免隨機性影響記憶一致性。  
- **手動筆記限制**：`shared/manual.md` 只作為「穩定規則、長期決策、常見限制、repo 習慣」的存放地，不能直接複製 Issue 原文。  

## Open Loops
- **等待 Issue 更新**：目前所有快照皆顯示「等待下一輪 issue 更新後再整理」，代表缺乏可供蒸餾的原始資料。  
- **未確定的長期規則**：若未來出現新 Issue，需檢視是否能形成新的穩定規則或限制。  
- **潛在主題探索**：持續觀察是否會有跨 Issue 的主題浮現，尤其是標籤 (`Top Labels`) 為 `none` 的情況下。
