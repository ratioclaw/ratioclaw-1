# Repository Memory

## Stable Context
- **長期規則與決策**  
  - 只將 **GitHub Issue / Comment** 視為原始事實來源，任何摘要或記錄皆須以此為根基。  
  - **Shared Manual Notes** 為人工維護的長期記憶，裡面的內容（如穩定規則、長期決策、常見限制、repo 習慣）屬於永久性上下文，除非有人手動更新，否則不會被自動覆寫。  
  - **compact‑memory workflow** 會讀取此手動筆記，但不會自行修改它。  

- **repo 習慣**  
  - 不要直接複製 Issue 原文到長期記憶檔案；只保留必要的概念與結論。  
  - 每日快照只負責 **蒸餾**（distill）跨 Issue 的主題、決策與未完成事項，若無可用 Issue，則保留既有記憶不變。  

- **資訊更新機制**  
  - 當有新的 Issue 產生或既有 Issue 被更新，日誌會在下一次快照時重新整理。  
  - 若快照期間 **沒有可用 Issue**，則本文件保持不變，僅保留先前已確定的穩定上下文。  

## Recent Themes
> 目前的每日快照（2026‑06‑23 至 2026‑06‑29）皆顯示「本次整理視窗沒有可用 issue」，因此 **沒有可辨識的跨 Issue 主題**。  
- **暫無重複出現的主題**。  

## Constraints
1. **資料來源限制**  
   - 只能從 Issue / Comment 抽取事實；手動筆記僅作為輔助說明。  
2. **內容呈現限制**  
   - 不得在長期記憶檔案中直接貼上 Issue 原文。  
   - 必須以 **工程可讀性**（清晰的 Markdown 結構、簡潔的敘述）呈現。  
3. **更新頻率**  
   - 只有在 Issue 狀態變化或新 Issue 出現時才會觸發內容變更。  
4. **不確定資訊處理**  
   - 若資訊僅在單一天出現且未形成穩定事實，必須放入 **Open Loops**，不應寫成規則。

- **未確定的長期規則**
