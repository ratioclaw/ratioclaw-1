# Repository Memory

## Stable Context
- **資料來源原則**  
  - 只從 GitHub Issue / Comment 取得原始事實，手動筆記僅作為長期規則與限制的彙整。  
  - `shared/manual.md` 為唯一的手動維護檔案，永遠不會被自動覆寫。  

- **記憶整理流程**  
  - 每日快照會由各 Issue Agent 彙整當天活躍的 Issue，若無可用 Issue，則保留既有記憶。  
  - `compact‑memory` 工作流會讀取本檔（`MEMORY.md`）與 `shared/manual.md`，作為長期上下文供所有 Agent 參考。  

- **Repo 習慣**  
  - **不複製** 原始 Issue 文字；只抽取概念、決策、限制與未完成事項。  
  - **蒸餾** 重複出現的資訊成為可重用的長期上下文。  
  - **標示不確定**：若資訊不足或相互矛盾，必須在本節或相關節點明確說明。  

- **長期決策與限制（目前已知）**  
  - 以 Issue 為唯一事實來源，手動筆記僅保存「穩定規則」與「常見限制」。  
  - 任何新規則或限制必須先在 Issue 中討論、取得共識，才能寫入 `shared/manual.md`。  

## Recent Themes
> 近期（過去 30 天）未偵測到跨 Issue 的共通主題或重複出現的議題。  
> - 所有每日快照皆顯示「本次整理視窗沒有可用 issue」，因此目前沒有可供蒸餾的主題。  

## Constraints
1. **資訊完整性**  
   - 必須保證本文件僅包含已在 Issue 中確認的事實或已在 `shared/manual.md` 中明確記錄的規則。  
2. **避免冗餘**  
   - 不得直接搬錄 Issue 標題或內容，僅保留概念層面的摘要。  
3. **不確定性標示**  
   - 若缺乏足夠證據支撐某項資訊，必須在相應節點加上「⚠️ 不確定」標記。  
4. **更新頻率**  
   - 每日快照完成後，若出現新 Issue，應立即在本文件的 **Open Loops** 或 **Stable Context** 中加入相應條目
