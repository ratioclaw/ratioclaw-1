# Repository Memory  

## Stable Context  
- **資料來源原則**：所有可追溯的事實與決策必須以 GitHub Issue 或 Comment 為唯一原始來源。  
- **手動筆記角色**：`shared/manual.md` 只保存 **長期穩定規則、決策、限制與 repo 習慣**，不會被自動流程覆寫，也不應直接複製 Issue 原文。  
- **compact‑memory 工作流程**：會在每次執行時讀取 `shared/manual.md`，但不會改寫它；因此手動筆記是 **唯一的長期記憶入口**。  
- **Issue 整理範圍**：每日快照只會處理最近 30 天內、狀態為 *all*、上限 100 件的 Issue。若當天無可用 Issue，系統會保留既有記憶，不產生新條目。  

> **不確定性**：目前沒有任何具體 Issue 被記錄，故無法從實務案例中抽取額外的穩定規則或流程。  

## Recent Themes  
- **無可辨識跨 Issue 主題**：2026‑06‑21 至 2026‑06‑27 的每日快照皆顯示「目前沒有可辨識的跨 issue 主題」。  
- **持續缺乏可用 Issue**：連續七天（2026‑06‑21~2026‑06‑27）系統皆回報「本次整理視窗沒有可用 issue」，顯示近期 repo 內缺乏活躍的待處理工作項目。  

## Constraints  
1. **不複製完整 Issue 文字**：手動筆記與長期記憶必須以摘要或規則形式呈現，避免冗長原文。  
2. **只能從 Issue/Comment 抽取事實**：若資訊僅出現在非 Issue 來源（如聊天、文件），則不列入正式記憶。  
3. **每日快照上限**：每次整理最多處理 100 件 Issue，且僅限最近 30 天內的資料。  
4. **手動筆記不可被自動覆寫**：任何自動化流程必須尊重 `shared/manual.md` 的寫入權限。  

## Open Loops  
- **等待新 Issue 出現**：目前所有每日快照皆顯示「等待下一輪 issue 更新後再整理」，因此任何未完成的工作、決策或跨 Issue 主題仍處於待定狀態。  
- **未確定的長期規則**：若未來出現新的 Issue，需評估是否能形成新的穩
