# Repository Memory

## Stable Context
- **長期規則**  
  - 只將 **GitHub Issue / Comment** 視為原始事實來源，任何摘要或記錄皆須以此為根基。  
  - **不直接複製** 完整的 Issue 內容於本檔；僅保留概念、決策與持續性的規則。  
  - 共享的手動筆記 (`shared/manual.md`) 為 **唯一的長期記憶入口**，其內容會被所有 agent 共同遵守。  

- **Repo 習慣**  
  - 每日快照 (`daily/*.json`) 由各 issue agent 產出，若當日無可用 Issue，則保留既有記憶，不新增條目。  
  - **跨 Issue 主題**、**決策** 與 **Open Loops** 必須在快照中明確標示，缺失時視為「無新資訊」。  

- **決策流程**  
  - 任何跨 Issue 的決策必須在每日快照的 **Decisions** 區塊記錄，未出現即代表目前無新決策。  

## Recent Themes
- **缺乏可用 Issue**：連續 7 天（2026‑06‑24 至 2026‑06‑30）皆未偵測到可用 Issue，系統持續回報「先保留既有記憶」。  
- **無跨 Issue 主題**：每日快照皆顯示「目前沒有可辨識的跨 issue 主題」。  
- **無新決策**：同樣的，所有日期的 **Decisions** 區塊皆為空。  

> 以上主題屬於近期觀測，顯示目前 repo 處於靜止或缺乏活躍議題的狀態。

## Constraints
1. **資料來源限制**  
   - 只能從 `daily/*.json` 與 `shared/manual.md` 兩類檔案抽取資訊。  
   - 任何未在 Issue 中出現的資訊，除非已被手動筆記確認，皆視為不可靠。  

2. **內容呈現限制**  
   - 本文件 **不作為索引頁**，亦不直接呈現原始 Issue 文本。  
   - 必須以 **工程可讀性**（Markdown 結構、清晰小節）呈現，同時保留 **龍蝦語氣**（溫暖、守護式的敘述）。  

3. **不確定性處理**  
   - 若資訊不足或相互矛盾，必須在相應小節標註「**不確定**」或「**缺乏資料**」。
