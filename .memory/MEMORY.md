# Repository Memory

## Stable Context
- **記憶管理流程**  
  - 以 GitHub Issue 為唯一原始資料來源，所有長期記憶皆應從 Issue 的內容、標籤與評論中抽取。  
  - `compact‑memory` 工作流會讀取本檔（`shared/manual.md`）作為「手動維護」的參考，但不會覆寫它。  
  - 每日快照（`daily/*.json`）負責把最近 30 天內的 Issue 活動濃縮成跨 Issue 主題、決策與未完成事項。  

- **手動筆記的角色**  
  - `shared/manual.md` 用於保存 **穩定規則、長期決策、常見限制** 以及 **repo 內部慣例**。  
  - 內容不應直接複製 Issue 原文，而是以精煉、可重用的敘述方式呈現。  

- **repo 習慣**  
  - 所有新功能、錯誤修復或流程變更必須以 Issue 形式提出，並在完成後關閉。  
  - 標籤 (`label`) 用於快速分類主題；若無標籤則視為「未分類」的臨時事項。  
  - 每日快照若檢測不到可用 Issue，則保留既有記憶，不產生新條目。  

## Recent Themes
- **持續無 Issue 活動**  
  - 2026‑07‑06 至 2026‑07‑12 的每日快照皆顯示「本次整理視窗沒有可用 issue」，說明近期（至少一週）內 repo 沒有新開或未關閉的 Issue。  
  - 這可能代表開發暫緩、需求已完成，或是 Issue 被其他渠道（如 PR、外部追蹤系統）取代。  

- **等待新 Issue**  
  - 每日快照的「Open Loops」皆寫著「等待下一輪 issue 更新後再整理」，顯示目前的未完成事項僅是「缺乏新資料」而非具體任務。  

> **備註**：上述主題僅基於最近七天的快照，若未來出現新 Issue，相關主題將重新評估並加入此節。

## Constraints
1. **資料來源限制**  
   - 只能從 GitHub Issue / Comment 取得事實，任何未在 Issue 中出現的資訊皆視為不確定。  
2. **避免原文複製**  
   - 本文件
